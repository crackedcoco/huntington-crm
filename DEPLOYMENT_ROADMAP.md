# Deployment & Multi-User Readiness Roadmap

> Browser audit completed 2026-04-13. Backend is multi-tenant ready; frontend needs work.

## Current State

**Backend (ready):**
- DynamoDB single-table with automatic tenant isolation (`TENANT#{tid}#ENTITY`)
- Cognito JWT auth with `custom:tenant_id`, `custom:plan`, `custom:role`
- Plan-based feature gates: starter (100 contacts), growth (500), scale (unlimited)
- Per-tenant rate limiting (100 req/min with DynamoDB atomic counters)
- Per-user usage tracking (tokens, costs, tool calls, model breakdown)
- Full REST API: CRUD contacts, audits, interactions, conversations, briefings

**Frontend (not ready):**
- Static `index.html` reading from `data.json` (generated by `scripts/export_data.py`)
- No authentication UI
- No API integration
- No tenant isolation on frontend

**Test Suite:** 15/43 pass, 28 fail (data tests crash — `CRM_DATA` null in test context)

## P1 — Required for multi-user deployment

- [ ] **P1.1 — Auth UI (login/signup/reset)**
  - Build login page against Cognito
  - Token management (store, refresh, logout)
  - Protected route wrapper

- [ ] **P1.2 — Connect frontend to Lambda API**
  - Swap `data.json` reads for live API calls with auth headers
  - Loading states, error handling
  - This is the single biggest task

- [ ] **P1.3 — CDK infrastructure stack**
  - `infrastructure/stacks/` is TODO
  - Need: DynamoDB table, Lambda, API Gateway, Cognito user pool, S3/CloudFront
  - All IaC'd and deployable

- [ ] **P1.4 — CORS configuration**
  - Lambda handler needs CORS headers for cross-origin API calls
  - `options_preflight()` exists but may need domain whitelisting

- [ ] **P1.5 — Tenant provisioning**
  - Signup → Cognito user → tenant_id assignment → plan selection

- [ ] **P1.6 — Role-based access on frontend**
  - Backend has `role` field but no frontend enforcement
  - Define: who sees what (admin vs member)

- [ ] **P1.7 — Contact detail view**
  - API: `GET /api/contacts/{id}` returns contact + audits + interactions
